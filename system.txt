第一章：核心战略与量化目标
（内容保留原意，仅对若干措辞/名词在全局范围内做了统一 — 见后文 YAML 字段统一说明）
核心战略 (Core Strategy)本系统的根本战略是构建一个基于 OpenClaw 的全自动记忆读写闭环。摒弃单次超长文本喂模的低效模式，利用 OpenClaw 支持外部 API 与长上下文处理的机制，打造“提取核心上下文 → 按大纲生成正文 → 拟真自我纠错 → 提取关键变动并覆写数据库”的循环系统。通过动态全局知识库（RAG）与角色档案持久化，从技术物理层面上杜绝长篇连载最致命的“吃书”与“OOC”问题。
核心量化目标
自动化闭环率：系统在注入初始故事大纲后，应能自动完成“检索-生成-校验-更新记忆”的完整步骤。
OOC 与漏洞控制：依托每次生成前强制读取的 YAML 档案与 SQLite 历史事件，角色行为偏离度 ≤ 1%，矛盾率目标尽可能趋近 0。

第三章：商业化内容与世界观执行标准（重写 —— 通用化变量驱动）
核心修正说明：将原文中所有硬编码的 IP/题材描述（例如具体“失眠社畜”“香肠派对手游”挂钩等）全部移除或替换为系统级变量。系统执行时严格读取这些变量作为唯一约束来源。
在自动化闭环运行期间，OpenClaw 必须严格遵守以下底层内容约束规则，任何偏离将触发系统的自动纠错打回：
故事主线锚点（Logline Anchor）系统必须在每次生成章节时强制读取 {Logline}（一句话核心大纲）作为最高优先级叙事锚点。所有章节剧情必须围绕该核心目标推进，不得出现与主线目标无关的长期偏离剧情。
题材红线控制（Genre Rules Enforcement）系统必须严格遵守 {Genre_Rules} 中定义的题材限制与禁用元素。Critic Agent 在审查阶段将自动检测违禁元素，一旦发现违反规则的内容（如题材不符、出现禁用科技或设定），系统将自动触发重写流程。
情绪与悬念回路（Narrative Tension Loop）每一章节必须在结构上满足商业连载小说的情绪推进规则：铺垫 → 冲突 → 情绪峰值 → 悬念断章。Beat Scheduler 将根据 {Core_Hook_Mechanic} 自动判断何时触发高潮或关键剧情反转。
社群传播强化机制（Meme Injection Layer）在剧情生成阶段，系统允许通过热点语义模块自动引入具有传播潜力的情节元素，例如：高情绪冲突、反常识设定、话题性人物行为或可被读者二次创作的“梗式桥段”。该模块为可选增强组件，不依赖任何特定 IP 或世界观。
备注：原文中“失眠”“香肠派对手游”等具体表述已全部删除，任何需要具体题材或 IP 的情形应当通过 {Logline} 或 {Genre_Rules} 注入（用户可在冷启动时定义）。

第五章（V3.1 新增）世界状态图 (World State Graph)
（注：此模块为 V3.1 新增，用于结构化存储世界实体与它们的关系，避免长篇中出现世界观冲突或地理/势力错误。）
5.1 世界节点（Node）
每个重要实体抽象为 Node（城市、势力、地点、机构、特殊遗迹等），以 JSON / Graph 表示，存储在 world_graph_edges 表与辅助 JSON 文件中。
示例 Node：
{
  "node_id": "loc_001",
  "type": "city",
  "name": "黑雾城",
  "attributes": {
    "population": 120000,
    "faction_control": "shadow_guild",
    "danger_level": 4
  }
}
5.2 世界关系（Edge）
节点间关系以 Edge 表示（统治 / 贸易 / 敌对 / 盟约 / 受灾等），关系包含强度与最后更新时间。
示例 SQL 建表语句：
CREATE TABLE world_graph_edges (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    source_node TEXT,
    relation_type TEXT,
    target_node TEXT,
    relation_strength INTEGER DEFAULT 1,
    last_updated_chapter INTEGER
);
5.3 使用场景
章节生成前 Query 相邻节点与关系，注入 Prompt，避免“城市归属错误”“阵营突兀合作”等逻辑失误。
第六章（V3.1 新增）角色关系图 (Character Graph Memory)
（注：记录角色之间动态信任/敌意/亲属/师徒/爱恨关系，作为生成决策的重要输入。）
6.1 关系表结构
CREATE TABLE character_relationships (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    char_a TEXT,
    char_b TEXT,
    relation_type TEXT,
    trust_level INTEGER DEFAULT 0,
    hostility_level INTEGER DEFAULT 0,
    last_updated_chapter INTEGER
);
关系类型示例：ally, enemy, mentor, family, romance, rival
6.2 更新规则
 每章结束后，系统不会直接从自由文本中自动推断角色关系变化。
  Writer Agent 在生成章节后，会调用结构化提取任务，
  要求模型返回严格 JSON 格式的事件摘要。
  LoreKeeper Agent 只负责解析该 JSON 数据，
  并根据其中的 relationship_updates 字段更新数据库。
  示例 JSON：
  {
    "events":[
      {
        "type":"relationship_change",
        "characters":["hero","mentor"],
        "relationship_updates":[
          {"char_a":"hero","char_b":"mentor","trust_delta":1}
        ]
      }
    ]
  }
  程序将验证 JSON 结构后再更新 character_relationships 表。
  不允许直接根据小说文本自动推断关系变化。
6.3 生成调用
Writer Agent 在生成时会读取角色关系摘要（例如“主角 — trust 4 — 导师；主角 — hostility 3 — 反派”）并据此约束行为与对话风格。
第七章（V3.1 新增）剧情节奏调度器 (Beat Scheduler)
（注：用于保证商业连载的节奏与爽点分布，支持可配置节奏模板与自动增强 Prompt。）
7.1 默认节奏（可配置）
story_beats:
  hook: {type: "fixed", value: 1}
  first_payoff: {type: "range", min: 2, max: 4}
  mini_climax: {type: "range", min: 8, max: 12}
  arc_climax: {type: "range", min: 25, max: 35}
  world_reveal: {type: "range", min: 80, max: 120}
Beat Scheduler 会根据配置区间自动选择合理章节，
避免固定节奏导致故事结构僵化。
7.2 Beat Scheduler 行为
在生成章节时，beat_scheduler.get_required_beat(chapter_num) 返回本章必须完成的 beat（例如：hook / payoff / climax），并自动在 Writer Prompt 中注入相关约束（例如“本章必须包含冲突升级与情绪高潮”）。

第九章：AI 代理体系（升级）
原 Writer / Critic 两代理扩展为多代理协作体系，以便实现更精细的规划、节奏与设定一致性控制。
新增代理（V3.1）
Planner Agent：负责章节级规划、Beat 分配、伏笔安排，与 Beat Scheduler 联动。
Writer Agent：根据 Planner 提供的结构化大纲与上下文执行文本生成（调用 OpenClaw）。
Critic Agent：对稿件进行规则审查（OOC、题材红线、节奏与推进性检查），并返回详细反馈与分数。
LoreKeeper Agent：负责世界状态图与角色关系图的同步更新、设定冲突检测，并在检测到高风险冲突时阻断生成或回滚修改。
生成流程（代理协作）
Planner 生成本章结构（包含 beat tag、关键事件点）
Writer 以 {Logline}、characters YAML、world_graph、character_graph 及最近事件为上下文生成初稿
Critic 审查初稿；不合格则返回修改意见给 Writer（自动或人工干预）
LoreKeeper 在定稿后解析并更新 YAML / Graph / SQLite，同时触发伏笔回收或新增
注意：所有关系变化与世界状态更新必须由
结构化 JSON 输出触发。
系统不会直接从小说文本自动推断复杂关系。

常见问题（补充）
Q: 为什么生成的文本仍然出现特定 IP 风格？A: 检查 config/world_setting.yaml 中 logline 或 prompts 是否包含了该 IP 的字词。系统不再在底层硬编码 IP；所有题材来自 {Logline} 与 {Genre_Rules}。
Q: 为什么 YAML 更新失败或解析报错？A: 请确保所有 YAML 文件使用的字段为 physical_health 和 mental_state（而不是旧的 physical_status / psychological_status）。
Q: 系统大量并发生成导致 OpenClaw 崩溃？A: 请调整 openclaw_api.py 中的 max_retries 与 _truncate_context 策略；并在 orchestrator 中限制并发度。

def generate_chapter_pipeline(chapter_num: int, outline_node: str):
    world_config = load_yaml("config/world_setting.yaml")
    planner = PlannerAgent(...)
    beat = planner.request_beat(chapter_num)
    characters = load_active_characters(outline_node)
    # 加载 world_graph / character_graph
    recent_events = db.query("SELECT event_summary FROM events_log ORDER BY id DESC LIMIT 3")
    writer_prompt = build_prompt(world_config, characters, recent_events, outline_node, beat)
    draft_text = openclaw_api.generate_with_retry(system_prompt, writer_prompt)
    critic_result = critic_agent.evaluate(draft_text)
    if critic_result.score < threshold:
        draft_text = openclaw_api.generate_with_retry(system_prompt, writer_prompt + "\n修改意见：" + critic_result.feedback)
    update_data = openclaw_api.extract_structured_data(draft_text)
    # LoreKeeper 解析并写入 YAML / Graph / SQLite
    lorekeeper.update_world_and_relations(update_data)
    return draft_text
